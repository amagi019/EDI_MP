# EDIシステム 要件定義書

## 1. システム化の目的・背景

- **現状の課題**: 毎月の注文書・注文請書の作成とメール送付、および取引先からの請求書回収に伴う事務負担の増大。また、稼働報告書ファイルの手動連携やGoogle Driveへの手動アップロードなど、手作業による煩雑な業務プロセスが存在。
- **解決策**: 取引先ポータル（EDIシステム）を構築し、帳票発行と回収を自動化・セルフサービス化する。外部API（Google Drive）とのシームレスな自動連携機能を提供する。
- **目標**: 事務工数の大幅削減、ヒューマンエラーの撲滅、ペーパーレス化の促進、電子帳簿保存法およびインボイス制度への完全対応。

## 2. 機能要件

| 区分     | 機能名             | 内容                                                                 |
|----------|--------------------|----------------------------------------------------------------------|
| 認証管理 | 取引先ログイン     | 取引先ごとに発行したID/PWで認証。Google Authenticator 等を用いたセキュリティ強化対応。 |
| 認証管理 | アカウント管理     | 取引先の担当者追加・削除、権限（閲覧のみ/DL可など）の設定。            |
| 発注管理 | 注文書・請書自動発行| 契約条件からPDFを自動生成。動的レイアウト（印紙枠、角印の自動配置や折り返し）表示機能。 |
| 発注管理 | 注文請書 回収      | 取引先がDLした際、または「承諾」ボタン押下で「受領済」ステータスに更新。 |
| 稼働管理 | 稼働報告書アップロード | パートナーが月間の稼働報告書（Excel等）をEDI上に直接アップロードする機能。 |
| 稼働管理 | Google Drive自動連携 | アップロードされた稼働報告書を、取引先（クライアント）共有用のGoogle Driveへ自動保存し、共有リンクを自動取得する機能。 |
| 請求管理 | 支払通知・請求書生成 | 自社で計算した確定金額をもとに「支払通知書」および「請求書（パートナー発行扱い）」のPDFを双方へ自動生成。 |
| 請求管理 | 長文自動レイアウト | 業務内容や案件名が長文になった場合、PDF出力時に枠内で自動改行・自動高さ調整を行う機能。|
| 請求管理 | 請求データDL       | 取引先が自社の会計処理用にXML/JSON形式でデータを取得できるAPIおよびCSV対応機能。 |
| 通知機能 | メールテンプレート管理 | システム内から送信される全メール（注文書発行、稼働報告、請求確定等）の件名・本文をDB管理し、管理者が画面から柔軟に変更できる機能。 |
| 通知機能 | 送信前プレビュー   | クライアント等への発信前にメール文面・添付ファイル・共有リンクをプレビューし、内容を編集してから送信できる機能。 |
| 管理側   | 進捗管理ダッシュボード | どの取引先がいつ帳票をDLしたか、稼働報告書が未提出の会社はないかなどの全体把握画面。 |

## 3. 業務フロー（To-Be）

### 3.1 発注・契約フロー
1. **発注**: 自社システムで注文内容を設定 → EDI上に「注文書」PDFを動的生成 → 取引先へ自動メール送信。
2. **受領**: 取引先がEDIにログインし、注文書を確認。「承諾」のアクションにより、EDIシステム上で「注文請書」が合意・保存される。

### 3.2 稼働報告・請求フロー
1. **稼働報告提出**: 取引先（パートナー）が月末〜月初にかけて、EDI画面から対象月の稼働報告書（Excelファイル）をアップロード。
2. **Drive自動連携**: EDIシステムが稼働報告書を受け取ると、バックグラウンドでGoogle Driveの指定フォルダへ自動アップロード。
3. **クライアント報告**: 自社管理者が「クライアント送付プレビュー」を確認。自動取得されたDrive共有リンクを含めた定型メールをワンクリックでクライアントへ送信。
4. **請求・支払確定**: 稼働時間を元に自社の計算処理を経て、検収金額が確定。
5. **PDF表示・ダウンロード**: 取引先（パートナー）はマイページから、自社宛の「請求書PDF」と自社からの「支払通知書PDF」の両方をブラウザ上でインライン確認、またはダウンロード可能。
6. **保存**: 全ての電子データは、電子帳簿保存法の要件に則りシステム内に法定期間保存される。

## 4. 非機能要件

- **セキュリティ**: SSL通信の暗号化。サービスアカウントを利用したGCP/Google Drive連携（セキュアなJSONキー管理）。
- **可用性**: 非同期/同期処理を適切に設計し、月末・月初等のアクセス集中時やGoogle APIの応答遅延時でもシステム全体がフリーズしない構造。
- **拡張性（DDDアーキテクチャ）**: ドメイン駆動設計（DDD）を採用し、モデル層・サービス層・プレゼンテーション層を明確に分離。将来的なAPI改修や機能拡張に耐えうる堅牢なアーキテクチャ。
- **法対応**: 電子帳簿保存法およびインボイス制度（適格請求書発行事業者番号の印字）の要件を完全網羅。
