# EDI_MP プロジェクト ベストプラクティス

本ドキュメントは、このプロジェクトの開発・保守における設計原則と必須ルールを定義する。
新機能の追加や既存機能の修正を行う際は、必ずこのガイドラインに従うこと。

---

## 1. フロントエンド・バックエンドの2層防御

### 原則
> **フロントでどれだけ安全に見えても、バックエンド側では常に「入力はすべて悪意がある可能性がある」と想定して実装する。**

### ルール

#### フロントエンド（第1層：UX改善）
- フォーム送信ボタンは、1回クリック後に **即座に非活性化** し、スピナーを表示する
- JavaScriptによるバリデーションはUX向上のために行うが、セキュリティの保証には使わない
- `base.html` のグローバルスクリプトが全POSTフォームを自動保護している

#### バックエンド（第2層：セキュリティ保証）
- **すべてのPOSTリクエスト**にCSRFトークン検証を適用する（Django標準）
- メール送信やステータス変更を伴うアクションには **冪等性チェック** を必ず実装する
  - 例：`if order.status == 'APPROVED': return`（既に承認済みなら処理スキップ）
- ユーザー入力は必ずサーバーサイドでバリデーションする
- 権限チェック（`is_owner_of_partner`, `StaffRequiredMixin`等）をビューに必ず実装する

## 2. ハードコーディングの防止

### 原則
> **会社固有の情報や可変データは、コード内にハードコーディングせず、必ずデータベースまたは環境変数で管理する。**

### ルール

#### メールテンプレート
- **【必須ルール】メールの雛形は、必ずメールテンプレート（`EmailTemplate`モデル）で修正可能にすること。**
- コード内（`views.py`など）にメールの件名や本文を直接ハードコーディング（固定書き）しない。
- 送信者名・会社名は `CompanyInfo` モデルから動的に取得する。

#### PDF出力
- 契約書・注文書・請求書の条文やテンプレートは、DBモデルで管理し管理画面でメンテナンス可能にする
- 会社名・住所・代表者名等の情報は `CompanyInfo` モデルから取得する
- フォールバック値にも会社固有情報を入れない（空文字列を使用）

#### 画像（印影・ロゴ等）
- 印影画像はコードに埋め込まず、`CompanyInfo.stamp_image` として管理画面から登録する
- デプロイスクリプトで画像ファイルを自動コピーしない（移管時のセキュリティリスク）
- `media/` ディレクトリの内容はGit管理対象外（`.gitignore`）

#### 設定値・認証情報
- APIキー、メールアドレス、接続先情報は **環境変数**（`.env`ファイル）で管理する
- `settings/base.py` のデフォルト値に本番の認証情報を入れない
- サービスアカウント情報は環境変数 `GOOGLE_DRIVE_SERVICE_ACCOUNT` で管理

## 3. アプリケーション移管への備え

### 原則
> **このアプリケーションは他社に移管される可能性を常に考慮し、会社固有情報を分離しておく。**

### チェックリスト（移管前の確認事項）
- [ ] `CompanyInfo` の全フィールドが正しく設定されているか
- [ ] `EmailTemplate` が移管先企業用に更新されているか
- [ ] `.env` ファイルが移管先環境に合わせて設定されているか
- [ ] `media/stamps/` の印影画像が移管先のものに差し替えられているか
- [ ] Google Drive連携のサービスアカウントが移管先のものに変更されているか
- [ ] Git履歴に機密情報が残っていないか

## 4. コーディング規約

### Djangoテンプレート
- `{{ ... }}` タグは **必ず1行で記述する**（改行バグ防止）
- 複雑なフィルター処理はテンプレートではなく `views.py` で計算する
- テンプレート作成・編集後は改行バグチェックを実施：
  ```bash
  grep -rn --include="*.html" $'\{{[^}]*$' .
  ```

### ビュー
- 重要な操作（承認・送信等）は `POST` メソッドを使用する
- すべてのビューに適切な権限Mixin（`LoginRequiredMixin`, `StaffRequiredMixin`等）を付与する
