# EDI・注文管理システム (EDI-MP)

## 1. 主な機能

- **進捗管理ダッシュボード**: ダッシュボード中心の業務運用。全タスクをパイプラインで一元管理。
    - **発注進捗パイプライン**: 注文書作成→承認→報告受領→請求書作成→送付→支払の6ステップ進捗表示。
    - **受注進捗パイプライン**: 受注→勤怠→請求書→入金の4ステップ進捗表示。
    - **期限管理**: 「あとX日」「X日超過」バッジ表示、担当者（自社/パートナー）の識別。
    - **クイックアクション**: 新規発注・新規受注・注文書一覧・支払通知等へのショートカット。
    - **一括操作**: 全タスク完了、個別タスクスキップ、支払完了の手動マーク。
    - **注文書リンク**: パイプライン詳細パネルから「📝注文書を見る」で注文書詳細に直接遷移。
- **パートナーオンボーディング**: 取引先による会社情報、インボイス登録番号、振込先情報のオンライン登録。
- **基本契約管理**: 契約書の自動生成、送付、オンライン承認、Google Drive自動保存。
- **デジタル発注フロー**:
    - **下書き (DRAFT)**: 管理者による発注データ作成。
    - **正式発行 (PUBLISH)**: パートナーへの公開・Google Drive保存。
    - **訂正再送付 (REPUBLISH)**: 発行済み注文書の訂正版PDF再生成・パートナーへ再送付。
    - **オンライン受領・承認**: パートナーによる内容確認と承認。
- **支払条件管理 (PaymentTerm)**: 締め日・支払月・支払日を構造化フィールドで管理。支払期限を自動計算。
- **請求書・支払通知書**: PDF生成、Google Drive保存。
- **クライアント管理（受注側）**: 取引先登録、受注管理、勤怠報告、請求書発行、入金管理。
- **月次タスク自動生成**: 発注確定時に期限付きタスクを自動作成、リマインドカレンダーで可視化。
- **電帳法・インボイス制度対応**:
    - 適格請求書発行事業者番号（T番号）の管理とバリデーション。
    - 注文書・注文請書のPDF永続保存および改ざん防止ハッシュの生成。
    - 確定日時のタイムスタンプ保持。
- **Google Drive連携**: 契約書・注文書・支払通知書を共有ドライブの専用フォルダに自動保存。

## 3. システムの起動方法

### ワンコマンドで起動（推奨）

```bash
./start.sh
```

このスクリプトは以下を自動で実行します:
1. 仮想環境の作成（初回のみ）
2. 仮想環境の有効化
3. 依存パッケージのインストール
4. データベースマイグレーション
5. 静的ファイル収集
6. 開発サーバー起動

> **初回実行時**: `chmod +x start.sh` で実行権限を付与してください。

### 手動で起動する場合

```bash
# 仮想環境の作成（初回のみ）
python3 -m venv .venv

# 仮想環境の有効化
source .venv/bin/activate

# 依存パッケージのインストール
pip install -r requirements.txt

# データベースマイグレーション
python manage.py migrate

# 開発サーバーの起動
python manage.py runserver 8090
```

### アクセスURL

| URL | 用途 |
|-----|------|
| http://127.0.0.1:8090/ | ローカル開発環境 |
| http://127.0.0.1:8090/admin/ | ローカル管理者画面 (Django Admin) |
| https://edi.macplanning.com/ | 本番環境 (QNAP TS-462) |
| https://edi.macplanning.com/admin/ | 本番管理者画面 |

## 4. 運用フロー

### パートナー管理（発注側）

1. **管理者登録** — CLI（`createsuperuser`）または `/signup/admin/` で管理者アカウントを作成
2. **パートナー登録** — ダッシュボード「パートナー新規登録」で会社名・メールアドレスを入力。招待メールが自動送信される
3. **オンボーディング** — パートナーが「会社情報を登録・更新する」からインボイス登録番号・振込先口座情報を登録
4. **基本契約** — 管理者が契約書PDFを生成・送付 → パートナーが「承諾」
5. **注文書 作成** — ダッシュボード「＋新規発注」→ 発注基本情報から注文書を作成（下書き → 正式発行）
6. **注文書 承諾** — パートナーが注文内容を確認し「承諾」（注文請書が自動生成・保存）
7. **稼働報告書 提出** — パートナーがExcelをアップロード → 解析結果を確認して確定
8. **請求書 作成** — 管理者が請求書を作成（パイプラインからスキップも可能）
9. **支払確認** — パイプラインの「💰支払完了」ボタンで手動完了
10. **訂正再送付** — 注文書詳細画面の「訂正再送付」ボタンでPDF再生成・メール再送（訂正が必要な場合）

### クライアント管理（受注側）

1. **取引先登録** — ダッシュボードのサイドバー「取引先登録」から顧客情報を登録
2. **受注登録** — ダッシュボード「＋新規受注」で受注基本情報を登録
3. **勤怠報告** — 稼働報告を作成・送付
4. **請求書発行** — 請求書を作成・送付（EDI顧客の場合はスキップ可能）
5. **入金確認** — パイプラインで入金確認、手動で完了マーク

### ダッシュボード中心の運用

- 全ての進捗はダッシュボードのパイプラインで確認可能
- 各画面から「← ダッシュボード」ボタンで即座に戻れる
- フィルタ: 未完了（デフォルト）、遅延、進行中、完了、全て
- 担当者表示: 「自」＝自社タスク、「P」＝パートナータスク

## 5. UI命名規則

ボタンやラベルの表記は以下のルールに従う。

### 用語の定義

| 用語 | 英語 | 定義 | 使用場面 |
|------|------|------|---------|
| **承諾** | Agree | 相手の依頼や要求を受け入れること（意思表示） | パートナーが管理者の提案を受け入れる操作 |
| **承認** | Approve | 組織の権限者が正しいと認め、許可すること（判断・権限） | 管理者が権限者として判断・許可する操作 |
| **確定** | Confirm | 自分の報告・入力内容を最終確認し、提出すること | パートナーが自分の報告内容を確定する操作 |

### ボタンラベルのルール

| ルール | 例 |
|--------|-----|
| ボタンラベルに「〜する」を付けない | ✅ 承諾 / ✅ 承認 / ❌ 承認する |
| 動作を表す名詞で統一 | 登録、送信、保存、削除、承諾、承認、確定、正式発行 |
| パートナーが相手の依頼を受け入れる操作 | **承諾**（基本契約、注文書、支払通知） |
| パートナーが自分の報告を確定する操作 | **確定**（稼働報告書） |
| 管理者が権限者として判断・許可する操作 | **承認**（請求書レビュー） |
| 2段階操作は動作名 + 「を実行」 | 差戻し → 差戻しを実行 |

## 7. Google Drive連携

契約書・注文書・支払通知書は Google Drive の共有ドライブに自動保存されます。

### フォルダ構成
| ドキュメント | 環境変数 |
|---|---|
| 契約書 | `GOOGLE_DRIVE_CONTRACT_FOLDER_ID` |
| 注文書 | `GOOGLE_DRIVE_ORDER_FOLDER_ID` |
| 支払通知書 | `GOOGLE_DRIVE_PAYMENT_FOLDER_ID` |

### 設定
- サービスアカウント: `edi-drive-uploader@edi-sophia-test.iam.gserviceaccount.com`
- キーファイル: `credentials/drive-service-account.json`（Git管理外）

## 8. データの永続性

開発・本番ともに PostgreSQL を使用しています。
データベースは Docker ボリュームに永続化されており、コンテナの停止・再構築でもデータは保持されます。

## 9. 技術仕様とコンプライアンス

- **データ完全性**: 承認された文書（PDF）はサーバー上に永続保存され、SHA256ハッシュが生成されます。
- **マスタ管理**: `BankMaster` による正確な金融機関データの選択をサポート。
- **外部連携 (SignatureService)**: `orders/services/signature_service.py` を通じて外部電子署名プロバイダーとの連携が可能。
- **Webhook受領**: `orders/webhooks.py` にて、外部サービスからの署名完了イベントを処理します。

## 10. 操作ログ

本番環境では、注文書・請求書・タスクの操作ログが `/app/logs/audit.log` に記録されます。

| 項目 | 内容 |
|------|------|
| ログファイル | `/app/logs/audit.log` |
| ローテーション | 日次（1日ごと） |
| 保持期間 | **30日分** |
| 対象ロガー | `orders`, `tasks`, `invoices` |
| ログレベル | INFO以上 |

### 記録される操作例

- 注文書の訂正再送付（PDF再生成・メール送信）
- タスク完了・スキップ処理
- 支払完了マーク
- メール送信（成功・失敗）

### ログ確認方法

```bash
# 最新のログを確認
ssh yutaka@192.168.50.198
cd /share/Container/EDI_MP
docker compose -f docker-compose.nas.yml exec web tail -50 /app/logs/audit.log
```

| コマンド | 実行タイミング | 内容 |
|---------|--------------|------|
