# EDIシステム 要件定義書

## 1. システム化の目的・背景

- **現状の課題**: 毎月の注文書・注文請書の作成とメール送付、および取引先からの請求書回収に伴う事務負担の増大。
- **解決策**: 取引先ポータルを構築し、帳票発行と回収を自動化・セルフサービス化する。
- **目標**: 事務工数の削減、ペーパーレス化の促進、電子帳簿保存法への対応。

## 2. 機能要件

| 区分     | 機能名             | 内容                                                                 |
|----------|--------------------|----------------------------------------------------------------------|
| 認証管理 | 取引先ログイン     | 取引先ごとに発行したID/PWで認証。初回ログイン時のパスワード変更強制。 |
| 認証管理 | アカウント管理     | 取引先の担当者追加・削除、権限（閲覧のみ/DL可など）の設定。           |
| 発注管理 | 注文書自動発行     | 基幹データに基づきPDFを自動生成。取引先がマイページからDL可能にする。 |
| 発注管理 | 注文請書 回収      | 取引先がDLした際、または「確認」ボタン押下で「受領済」ステータスに更新。|
| 請求管理 | 自社作成請求書     | 自社で計算した確定金額を「請求明細（兼 支払通知）」として掲載。         |
| 請求管理 | 請求データDL       | 取引先が自社の会計処理用にCSVやPDF形式でデータを取得できる機能。       |
| 通知機能 | 自動メール通知     | 注文書発行時や請求確定時に、登録アドレスへ通知メールを自動送信。       |
| 管理側   | 進捗管理ダッシュボード | どの取引先がいつ帳票をDLしたか、未確認の会社はないかの一覧表示。         |

## 3. 業務フロー（To-Be）

1. **発注**: 自社システムから注文データをインポート → EDI上に注文PDFを生成 → 取引先へ自動メール。
2. **確認**: 取引先がEDIにログインし、注文書をDL。システム側で「確認済」フラグを立てる。
3. **請求**: 月末に自社で検収金額を確定 → 請求（支払）明細を公開 → 取引先へ自動メール。
4. **保存**: 電子データとしてシステム内に法規定の期間保存。

## 4. 非機能要件

- **セキュリティ**: SSL通信の暗号化。特定IP以外からのアクセス制限（必要に応じて）。
- **可用性**: 月末・月初等のアクセス集中時でも遅延なく動作すること。
- **法対応**: 電子帳簿保存法およびインボイス制度の要件を満たす保存機能。
